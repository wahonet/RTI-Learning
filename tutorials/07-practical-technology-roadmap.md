# 实践路线：怎样把论文变成我们的微痕扫描系统

## 1. 我们到底要做什么

目标不是“做一个RTI查看器”，而是做一套面向汉画像石、碑刻、石经、甲骨等对象的微痕数字化流程。

最终它应该能输出：

- 原始多光照照片。
- 可交互重打光结果。
- 法线图。
- 虚拟掠射光图。
- 增强灰度图。
- 数字拓片。
- 初步线图。
- 标注和释读。
- 全信息对象包。

## 2. 三层技术架构

### 第一层：采集层

负责把真实文物变成可靠数据。

近期方案：

```text
手持H-RTI
固定光源矩阵
普通RGB相机
反光球或灯位标定
```

进阶方案：

```text
偏振片/偏振相机
近红外/多光谱
摄影测量或结构光3D扫描
```

### 第二层：计算层

负责从数据中提取微痕信息。

近期算法：

```text
RelightLab生成RTI
光度立体估计法线
虚拟掠射光
CLAHE局部增强
边缘检测生成线图
灰度/反相/阈值生成数字拓片
```

进阶算法：

```text
BRDF/法线优化
偏振法线估计
多光谱特征分离
Neural RTI
AI线图生成
```

### 第三层：研究层

负责让数据可用于学术研究和文博应用。

包括：

```text
对象档案
采集元数据
处理参数
标注
释读
版本管理
导出研究包
Web展示
```

## 3. 第一阶段实验：手持H-RTI

目标：理解RTI基本原理。

对象：
- 硬币。
- 刻字石片。
- 手工刻痕板。

采集：
- 36张左右。
- 手持LED。
- 反光球。
- 固定相机。

输出：
- RTI文件。
- 增强图。
- 法线可视化截图。

成功标准：
- 移动虚拟光源时，刻痕明暗随方向变化。
- 至少有一个角度比普通照片更清晰。
- 记录采集失败点。

## 4. 第二阶段实验：固定光源矩阵 + 光度立体

目标：进入真正的微痕复现路线。

对象：
- 刻字石片。
- 手工浅刻线。
- 小块石材样本。

硬件：
- 相机固定。
- 8到16个LED。
- LED位置可测量。
- 暗环境。

输出：
- 法线图。
- 虚拟掠射光图。
- 增强灰度图。
- 数字拓片初稿。

成功标准：
- 同一刻痕在法线图中有方向变化。
- 虚拟掠射光能稳定显示浅刻线。
- 数字拓片比普通照片更适合识读。

## 5. 第三阶段实验：数字拓片和线图

目标：把微痕增强结果转成研究者可用成果。

输入：
- 增强灰度图。
- 法线图。
- 虚拟掠射光图。

处理：
- 背景压制。
- 局部对比度增强。
- 边缘检测。
- 形态学连接。
- 人工校正。

输出：
- 黑白数字拓片。
- 灰度数字拓片。
- 初步线图。

成功标准：
- 能解释每一张拓片是怎么生成的。
- 参数可重复。
- 不把裂纹和石材颗粒误当作图像线条。

## 6. 第四阶段实验：全信息模型

目标：避免数据散乱，形成可研究、可复核、可展示的对象包。

对象包结构：

```text
object_id/
  raw_images/
  metadata/
  light_positions/
  rti_outputs/
  normals/
  enhanced/
  rubbings/
  line_drawings/
  annotations/
  processing_logs/
```

关键原则：
- 原始数据永远保留。
- 每个输出都有来源和参数。
- 标注绑定到具体图像版本。
- 允许多个释读版本并存。

## 7. 技术选择建议

### 现在就做

- H-RTI。
- 固定光源矩阵。
- 光度立体。
- 法线图。
- 虚拟掠射光。
- 数字拓片。
- 初步线图。

### 稍后做

- 3D扫描融合。
- 偏振成像。
- 多光谱/近红外。
- BRDF法线优化。

### 研究储备

- Neural RTI。
- AI线图生成。
- 半监督轮廓检测。
- 文化遗产知识图谱。

## 8. 为什么这样排

第一阶段必须可解释、可复现、低成本。固定光源矩阵和光度立体正好满足：

- 硬件能搭。
- 算法能解释。
- 输出对微痕直接有用。
- 与传统RTI兼容。
- 与公开资料中的“光源矩阵、微痕增强、电子拓片、线图”对应。

Neural RTI、多光谱、偏振成像都很有价值，但如果没有稳定的多光照数据和对象包管理，直接做这些会变成空中楼阁。

## 9. 下一步行动

建议下一步不是继续泛读，而是进入小实验设计：

1. 选一个小样本：硬币、刻字石片或手工刻痕板。
2. 设计手持H-RTI采集。
3. 设计8灯固定矩阵草案。
4. 用同一对象分别采集H-RTI和固定光源数据。
5. 对比普通照片、RTI增强、法线图、数字拓片。

完成这个实验，我们就能判断后续软件最小闭环该怎么写。
